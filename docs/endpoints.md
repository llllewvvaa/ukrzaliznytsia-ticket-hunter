# UZ API — Endpoint & Auth Discovery

> Result of **Step 1 (Discovery spike)**. Everything below was derived from a
> `booking.uz.gov.ua.har` (143 entries) + 4 HTML snapshots, extracted into `fixtures/*.json`.
> Items marked **TBD** require a *fresh* authenticated HAR (active search + login) that cannot be
> captured offline — see [Open items](#open-items--requires-fresh-har).

## Hosts

| Host | Role |
|---|---|
| `https://app.uz.gov.ua/api` | Real REST API (`BASE_URL_API` in the Nuxt config, confirmed in `main-state.html` / `train-decide.html`). |
| `https://booking.uz.gov.ua` | Nuxt 3 SPA frontend (Pinia + `piniaPluginPersistedstate`). Origin of all API calls. |
| `https://www.uz.gov.ua` | Public **static timetable** site (no auth). Used only to list scheduled trains **before sales open** for the pre-sale flow — see [Timetable](#timetable-static-schedule). |

## Timetable (static schedule)

Isolated in `lib/timetable.ts` (separate from `uz-api.ts`: different host, no booking.uz auth,
HTML not JSON). Read-only helper — callers degrade to manual train-number entry on failure.

| Endpoint | Response | Parser |
|---|---|---|
| `GET /passengers/timetable/suggest-station/?q={q}` | `text/plain` JSON: `["Назва~id,id,…", …]` | `parseSuggestStations` |
| `GET /passengers/timetable/?from_station={ids}&to_station={ids}&select_time=2&time_from=00&time_to=24&by_route=1` | HTML table (`#cpn-timetable`): № · маршрут · періодичність · станція/час відпр./приб. | `parseTimetableHtml` (fixture: `fixtures/timetable-route.html`) |

**Caveats:**
- `suggest-station` answers the literal **`false`** when nothing matches the query, and it
  substring-matches its *own* naming (`Київ-Пас.`, `Київ Місто`) — so a full booking.uz name
  like `Київ-Пасажирський` misses. The UI drives it the way the site does: the user searches &
  **picks** the timetable station (`TimetableStationInput`), and we feed the chosen id bundle into
  the timetable page. The station inputs are auto-seeded from the booking.uz names by widening the
  query down to the city stem (`stationQueryCandidates`) and picking the best hit (`pickBestStation`);
  `Київ` returns the `739,47125,…` city bundle (all terminals).
- Timetable station ids are **bundled per city** (`739,47125,…`) — a *different id space* from
  booking.uz's numeric station ids. We only extract a train **number**; the reserve still runs off
  the booking.uz route + that number.
- Number formats differ (timetable `"66"` vs booking.uz `"066Ш"`) → `matchTrip` compares the
  **numeric core** (`normalizeTrainNumber`).
- No JSON API → HTML scraping (parsed in pure functions, tested on a fixture). Treat as a soft
  helper; the SW can't `DOMParser`, so parsing is regex-based.

CORS: every API call is preceded by an `OPTIONS` preflight → `204`. `access-control-allow-origin`
echoes `https://booking.uz.gov.ua`, so an extension with `host_permissions` for `app.uz.gov.ua`
can call the API directly from the service worker.

## Auth headers

Confirmed on **every** `app.uz.gov.ua/api` request (values from a real session):

```
accept: application/json
x-client-locale: uk
x-session-id: 00000000-0000-4000-8000-000000000000      # UUID v4, generated by the SPA
x-user-agent: UZ/2 Web/1 User/1000001                   # User/<profile.id>
origin: https://booking.uz.gov.ua
referer: https://booking.uz.gov.ua/
```

- `x-user-agent` user id (`1000001`) === `profile.id` (account id), **not** `passenger.id`.
- **`Authorization` header**: its *value* is redacted in this HAR export, but the OPTIONS preflight
  advertises `access-control-request-headers: authorization,content-type,x-client-locale,x-session-id,x-user-agent`.
  → A `Authorization: Bearer <jwt>` is **assumed** but the exact Pinia/localStorage key holding the
  token is **TBD** (validated against a fresh HAR + live page inspection).
- **No cookies** are used for the API (consistent with non-credentialed CORS).

### Origin / Referer for SW requests (DeclarativeNetRequest)

`origin` and `referer` are **forbidden** request headers — the browser ignores any attempt to set
them via `fetch(..., { headers })`, so they are *not* in `buildHeaders` (`src/lib/auth.ts`). Requests
the **service worker** makes to `app.uz.gov.ua` therefore go out with the extension's own origin
(`chrome-extension://…`) instead of `https://booking.uz.gov.ua`, risking CORS/WAF rejection.

`src/lib/net-rules.ts` registers a **session-scoped** `chrome.declarativeNetRequest` `modifyHeaders`
rule (`installNetworkRules()` from `background.ts`) that rewrites them on the wire:

```jsonc
// permission: declarativeNetRequestWithHostAccess  (acts only on hosts we already hold)
action.requestHeaders = set {
  origin:         "https://booking.uz.gov.ua",
  referer:        "https://booking.uz.gov.ua/",
  sec-fetch-site: "same-site",  sec-fetch-mode: "cors",  sec-fetch-dest: "empty"
}
condition = { requestDomains: ["app.uz.gov.ua"], tabIds: [-1] }   // -1 = TAB_ID_NONE → SW only
```

- `tabIds:[-1]` is only honoured for **session** rules (not static/dynamic), hence
  `updateSessionRules` on startup (idempotent: `removeRuleIds` first). It also leaves the live
  booking.uz **tab's** own requests untouched.
- Only forbidden / `sec-fetch-*` headers are rewritten. `Authorization` / `x-session-id` /
  `x-user-agent` / `x-client-locale` stay in `buildHeaders` (dynamic per live session).
- **DNR does not bypass reCAPTCHA** — the content-script reserve stays as the challenge fallback.

## reCAPTCHA

reCAPTCHA is integrated **app-wide**: `https://www.gstatic.com/recaptcha/releases/.../recaptcha__en.js`
loads on *all* page snapshots, and challenge assets (`audio_2x`, `image_2x`, `refresh_2x`, `info_2x`)
appear in the HAR. The `POST /api/v4/orders` **request body in this HAR contains no reCAPTCHA token**
(see `fixtures/create-order-payload.json`), so whether `/orders` is reCAPTCHA-gated (token in a
header vs body, v2/v3/Enterprise, sitekey) is **TBD**. Architecture assumes it *may* be gated and
performs the reserve inside the page context to reuse the SPA's own token. (sitekey **TBD**.)

### `GET /api/stations?search={q}` (confirmed live)

Confirmed against the live API (`curl`, authenticated session). Used by the
«from»/«to» autocomplete in the new-hunt form.

```
GET https://app.uz.gov.ua/api/stations?search=су
→ 200, body is a *flat* array (no `data` wrapper):
[
  { "id": 2204450, "name": "Суми" },
  { "id": 2200145, "name": "Судилків" },
  ...
]
```

- Query param is **`search`** (not `query`); no `v2`/`v3` version prefix.
- Returns up to ~10 matches; same `{ id, name }` shape & ids as `station_from`/`station_to`
  in `GET /api/v3/trips/{id}` — so a picked station id feeds directly into search/trip calls.
- Requires the standard auth headers (`Authorization: Bearer`, `x-session-id`, `x-user-agent`,
  `x-client-locale`).

### `GET /api/v3/trips?station_from_id=&station_to_id=&with_transfers=0&date=` (confirmed live)

Confirmed against the live API (`curl`, authenticated session). This is the request fired by the
SPA after clicking «Знайти». Powers the monitor loop (`searchTrips` → `matchTrip`).

```
GET https://app.uz.gov.ua/api/v3/trips?station_from_id=2200001&station_to_id=2218000&with_transfers=0&date=2026-07-10
→ 200
{
  "station_from": "Київ-Пас",          // display label (string, not the {id,name} object)
  "station_to":   "Львів",
  "direct": [                            // ← the matchable trips
    {
      "id": 11378092,
      "depart_at": 1783646940, "arrive_at": 1783677720,
      "station_from": "Київ-Пас", "station_to": "Львів",
      "train": {
        "id": 57, "number": "021О", "type": 0,
        "station_from": "Харків-Пас", "station_to": "Львів",
        "wagon_classes": [
          { "id": "П", "name": "Плацкарт", "free_seats": 0, "price": 0, "amenities": [ ... ] },
          { "id": "К", "name": "Купе",     "free_seats": 0, "price": 79326, "amenities": [ ... ] }
        ]
      },
      "monitoring": { "allowed": true, "auto_purchase": true },
      "max_passengers_count": 4, "distance": 627, "seat_hold_enabled": true
    }
    // ... one entry per direct train
  ],
  "with_transfer": null,
  "monitoring": { "available_type": "trains", "active": null },
  "sale_start": null
}
```

- Query params: **`station_from_id`**, **`station_to_id`** (the same numeric ids returned by
  `/api/stations?search=`), **`with_transfers=0`** (we ignore transfer routes), **`date=YYYY-MM-DD`**.
- The matchable trips live under **`direct[]`** (`parseSearchTrips` returns exactly this array).
- Each `direct[i]` is shaped like `GET /api/v3/trips/{id}` **except** `station_from`/`station_to`
  are plain display strings (not `{id,name}`) and there are no `route_points`. `matchTrip` only
  reads `train.number` + `train.wagon_classes[].{id,free_seats}`, so the shape is compatible.
- Empty `direct: []` for a date with no trains on sale (e.g. outside the 45-day sale window).
- Top-level `monitoring.available_type` (`"trains"`/`"route"`) hints at the native monitoring feature
  (per-train vs per-route) — endpoint still **TBD**.

### `GET /api/v3/trips/{id}` (shape)

```jsonc
{
  "id": 3000001,
  "depart_at": 1784484060,            // unix seconds
  "arrive_at": 1784538300,
  "station_from": { "id": 2200001, "name": "Київ-Пас" },
  "station_to":   { "id": 2218218, "name": "Татарів" },
  "train": {
    "id": 5649, "number": "057К", "type": 0,
    "station_from": "Київ-Пас", "station_to": "Ясіня",
    "wagon_classes": [
      { "id": "П", "name": "Плацкарт", "free_seats": 3, "price": 26880, "amenities": [ ... ] },
      { "id": "К", "name": "Купе",     "free_seats": 0, "price": 40205, "amenities": [ ... ] }
    ]
  },
  "route_points": [ { "station": {id,name}, "arrive_at", "depart_at" }, ... ],
  "monitoring": { "allowed": true, "auto_purchase": true },
  "with_iron_km_program": false,
  "max_passengers_count": 4,
  "seat_hold_enabled": true
}
```

- `price` is in **kopecks** (`26880` = 268.80 UAH).
- `wagon_classes[].id` is the coach type code: `Л` (luxury), `К` (coupe), `П` (platzkart),
  `С1`/`С2` (seated). `free_seats` is the live availability we monitor.
- `monitoring.auto_purchase: true` ⇒ a **server-side** monitoring/auto-purchase feature likely
  exists (native mode candidate) — endpoint **TBD**.

### `POST /api/v4/orders` (request body — verified)

```jsonc
{
  "trip_id": 3000001,
  "reservations": [
    {
      "passenger_id": 2000001,
      "first_name": "...", "last_name": "...",
      "wagon_id": "eyJpdiI6...mac:...tag:",   // Laravel-encrypted blob — cannot be built locally
      "seat_number": 28,
      "input_type": null, "privilege": null, "privilege_data": {},
      "privilege_id": null, "companion_id": null,
      "services": [
        { "id": "bedding", "title": "Постіль", "price": 9500,
          "select_type": "checkbox", "select_units_max": null,
          "selected_by_default": true, "value": true, "details": { ... } }
      ],
      "passenger_options": [],
      "save_in_passengers": false
    }
    // ... one entry per passenger (sample: 2 passengers, seats 28 & 25)
  ]
}
```

- `wagon_id` is an **encrypted blob** → must be fetched fresh per reserve (cannot be precomputed).
- Response `429`: `{ "message": "Забагато спроб. Зробіть паузу та спробуйте ще раз пізніше.", "error_code": null }`.
- Success response shape (→ `cart_id`) **not present** in this HAR (the only POSTs captured were 429).

### `POST /api/trips/{tripId}/seats/hold` (confirmed live)

Fired by the SPA the moment a seat is clicked, **before** the order is created. Holds the seat for
the current session so `POST /orders` can reference it.

```
POST https://app.uz.gov.ua/api/trips/11287381/seats/hold
Content-Type: application/json
{ "seats": [ { "wagon_number": "3", "place": 20 } ], "uuid": "00000000-0000-4000-8000-000000000000" }
```

- `wagon_number` is the **plain** coach number as a string (e.g. `"3"`), **not** the encrypted
  `wagon_id` used in `/orders`; `place` is the seat number.
- `uuid` equals the `x-session-id` header (the live SPA session id).
- Implemented as `holdSeats(tripId, seats, uuid)`; the reserve executor calls it right after picking
  seats and before `createOrder`.

### `GET /api/v4/carts/{id}` (polling)

- `202` (queued) → `{ "cart_id", "title", "description", "retry_in": 6 }`; keep polling every `retry_in` sec.
- `200` (ready, confirmed live) → `{ "id", "amount", "expire_at", "orders": [...], "donation" }`
  (see `fixtures/cart-ready.json`). Note the cart is keyed **`id`** (not `cart_id`), and
  **`expire_at`** (unix seconds) is the reservation hold deadline shown as «Заброньовано на HH:MM».
  `parseCartStatus` falls back to `id` for `cart_id` and surfaces `expire_at` → `job.reservedUntil`.
  There is **no `payment_url`** — the user pays at the `/payment` SPA page (see SPA routes).
  Each `orders[].reservations[]` confirms the held seat (`wagon_number`, `wagon_class_id`,
  `seat_number`), so a real reserve looks like e.g. `6 вагон, 14 місце` + `16 місце`, Купе.

### `GET /api/v4/orders-with-routes` (confirmed live)

Current (unpaid / upcoming) orders. Powers the **active** section of the «Мої квитки» tab.

```
→ 200 { "tickets_version": 61, "has_archived": true, "orders": [ ...UserOrder ], "tickets_products": [] }
```

### `GET /api/v2/orders/archived?page=` (confirmed live)

Paginated booking history (the **archive** section of the «Мої квитки» tab).

```
→ 200
{
  "data": [
    { "id", "train": { station_from, station_to, number, type },
      "trip": { id, station_from, station_to, depart_at, arrive_at },
      "tickets": [ { id, note, returned_at, pdf_url, has_return_status,
        "reservation": { first_name, last_name, wagon_number, wagon_class_id,
          wagon_class_name, seat_number, services:[{id,title}], price, description } } ] }
  ],
  "meta": { "pagination": { total, count, per_page, current_page, total_pages, links } }
}
```

- `price` is in **kopecks**; a ticket with a non-null `returned_at` was refunded («повернено»).
- Parsed by `parseArchivedOrders` → `{ orders, pagination }`.
